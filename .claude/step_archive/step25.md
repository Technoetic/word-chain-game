# Step 25 - 코드 포매팅

프로젝트의 언어, 프레임워크, 기존 설정을 분석하여 적절한 포매팅 전략을 동적으로 결정한다.

## 작업 흐름

1. **프로젝트 분석**
   - 사용 중인 프로그래밍 언어 식별
   - 기존 포매터 설정 파일 확인 (.prettierrc, .editorconfig, pyproject.toml 등)
   - 프로젝트의 코딩 스타일 가이드 파악

2. **포매터 선택**
   - 기존 설정 파일이 있으면 해당 도구 사용
   - 없으면 언어별 표준 포매터 자동 선택
   - 여러 언어 혼용 시 각각 적절한 도구 적용

3. **포매팅 실행**
   - 선택된 포매터로 전체 코드베이스 포매팅
   - 변경 사항 검토 및 기록

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용하여 포매팅을 수행한다.

**포매팅 단계에서 절대로 plan mode를 사용하지 않는다.**

**객관적 검증 기준:**
- 모든 코드가 포매팅 규칙 준수 (포매팅 체크 실패 시 포매팅 실패)

**검증:**
- `.claude/hooks/formatting-validator.ps1`에서 자동 검증

**검증 실패 시:**
- 포매팅 실패 원인을 분석
- 코드를 재포매팅
- 검증 통과할 때까지 반복

**포매팅 결과는 청크 단위로 저장한다:**

```
step25_포매팅_chunk1.md (500줄 이하)
step25_포매팅_chunk2.md (500줄 이하)
step25_포매팅_chunk3.md (500줄 이하)
...
```

**작성 규칙**:
- 각 청크는 500줄 이하로 작성 (성능 최적화)
- `.claude/hooks/research-validator.ps1`에서 각 청크 검증 (BOM/CRLF/줄수/파일크기)
- 청크 그대로 유지 (병합 안 함)

서브에이전트는 항상 haiku를 사용한다.
