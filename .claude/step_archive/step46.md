# Step 46 - 격자 오버레이 디자인 검증

Step 45 디자인 시각 검증 완료 후, 격자(Grid) 오버레이를 적용하여 정밀한 정렬/간격 검증을 수행한다.

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**스크린샷 없이 통과 처리 금지. 반드시 Claude가 직접 스크린샷을 눈으로 확인한다.**

## 사전 분석 (동적 결정)

1. **프로젝트 디자인 시스템 분석**
   - 사용 중인 CSS 프레임워크/디자인 시스템의 기본 간격 단위(spacing unit) 파악
   - 예: Tailwind(4px), Material(8px), Bootstrap(4px/8px), 커스텀 등
   - 프레임워크가 없으면 실제 코드에서 가장 빈번하게 사용되는 간격 값 추출

2. **격자 간격 결정**
   - 분석된 디자인 시스템의 기본 단위를 격자 간격으로 사용
   - 디자인 시스템이 없으면 실제 CSS에서 추출한 최소 공통 간격 사용
   - 간격이 너무 작으면(2px 이하) 가독성을 위해 배수로 조정

3. **격자 색상/투명도 결정**
   - 프로젝트 배경색을 분석하여 격자선이 잘 보이는 색상 선택
   - 밝은 배경 → 어두운 격자선, 어두운 배경 → 밝은 격자선
   - 투명도는 UI 요소를 가리지 않으면서 식별 가능한 수준으로 조정

4. **검증 대상 결정**
   - Step 45에서 검증한 뷰포트/페이지 조합을 기반으로 격자 검증 범위 결정
   - 프로젝트에 실제 존재하는 레이아웃 패턴 기반으로 검증 항목 구성
   - 해당 없는 항목 제외 (예: 단일 컬럼이면 컬럼 정렬 검증 제외)

## 격자 오버레이 방식

1. **임시 CSS 삽입** — 사전 분석에서 결정된 간격/색상으로 격자 오버레이 CSS를 Playwright의 `page.addStyleTag()`로 런타임 주입한다 (소스 코드 수정 없음):
   ```css
   body::before {
     content: '';
     position: fixed;
     top: 0; left: 0; right: 0; bottom: 0;
     background: repeating-linear-gradient(
       to right, {색상} 0px, {색상} 1px, transparent 1px, transparent {간격}px
     ),
     repeating-linear-gradient(
       to bottom, {색상} 0px, {색상} 1px, transparent 1px, transparent {간격}px
     );
     pointer-events: none;
     z-index: 99999;
   }
   ```
2. **Playwright로 스크린샷 촬영** — 격자가 적용된 상태에서 스크린샷을 `.claude/screenshots/grid/` 에 저장
3. **소스 코드 오염 없음** — `page.addStyleTag()`로 주입하므로 별도 제거 작업 불필요

## 검증 관점 (해당하는 항목만 동적 선택)

- **수직/수평 정렬** — 요소들이 격자선에 맞게 정렬되어 있는지
- **간격 균일성** — 동일 역할 요소들의 간격이 격자 단위로 균일한지
- **여백 대칭** — 좌우/상하 여백이 격자 기준으로 대칭인지
- **기준선 정렬** — 텍스트 기준선이 격자에 맞는지
- **컬럼 정렬** — 그리드/컬럼 레이아웃이 격자에 정합하는지
- **컴포넌트 내부 일관성** — 같은 유형의 컴포넌트들이 동일한 격자 규칙을 따르는지

## 검증 절차

1. 사전 분석 결과를 기반으로 격자 간격/색상/대상 동적 결정
2. Step 45에서 사용한 뷰포트/페이지 조합에 격자 오버레이 적용
3. Playwright 스크립트로 격자 적용 스크린샷 촬영
4. Claude가 스크린샷을 직접 Read하여 격자 기준 정렬/간격 확인
5. **문제 발견 시:**
   - 어떤 요소가 격자에서 몇 px 벗어났는지 구체적으로 기록
   - 해당 소스 코드(CSS/스타일) 수정
   - 스크립트 재실행 → 스크린샷 재촬영
   - Claude가 다시 직접 확인
   - **통과할 때까지 무한 반복**
6. 모든 항목 통과 확인 후 다음 단계 진행

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다.

서브에이전트는 항상 haiku를 사용한다.
