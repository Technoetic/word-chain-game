# Step 53 - 접근성 테스트

프로젝트 유형, 타겟 사용자층, 법적/산업 요구사항, UI 특성을 분석하여 접근성 기준·검사 범위·도구를 동적으로 결정한다.

결정된 기준에 따라 자동화 접근성 테스트를 구현하고, 위반 0건이 될 때까지 수정을 반복한다.

**이 단계에서 절대로 plan mode를 사용하지 않는다.**

**위반 0건 확인 없이 통과 처리 금지.**

## 사전 분석 (동적 결정)

1. **프로젝트 유형 파악**
   - 웹 앱 / 데스크톱 앱 / 모바일 앱 / CLI 등
   - 프로젝트 유형에 적합한 접근성 검사 도구 선택
   - 웹 앱이면 `@axe-core/playwright` 사용 (Step 03에서 이미 설치됨)
   - 웹 앱이 아니면 해당 플랫폼의 접근성 검사 도구 조사 후 선택

2. **타겟 사용자·법적 요구사항 분석**
   - 프로젝트 대상 지역·산업의 접근성 법규 파악 (ADA, EN 301 549, 장애인차별금지법 등)
   - 적용해야 할 WCAG 수준 결정 (A / AA / AAA)
   - 산업 특수 요구사항 확인 (공공기관, 금융, 교육 등)

3. **UI 특성 기반 검사 범위 결정**
   - step09_설계_chunk*.md에서 UI 구조/컴포넌트 목록 추출
   - 프로젝트에 실제 존재하는 UI 요소 기반으로 검사 항목 구성
   - 해당 없는 항목은 제외 (예: 영상 없으면 자막 검증 제외, 폼 없으면 폼 라벨 제외)

4. **검사 도구 태그·규칙 결정**
   - axe-core 사용 시: 분석 결과에 맞는 withTags 조합 결정
   - 프로젝트에 불필요한 규칙은 disableRules로 제외
   - 필요 시 커스텀 규칙 추가

## 검사 관점 (해당하는 항목만 동적 선택)

axe-core 등 자동화 도구가 검사할 수 있는 영역:

- **색상 대비** — 텍스트/배경 대비 비율 기준 미달
- **대체 텍스트** — img alt 누락, 장식 이미지 구분
- **ARIA 속성** — role/aria-label/aria-describedby 정합성
- **폼 라벨** — input-label 연결
- **헤딩 순서** — 계층 구조 위반
- **키보드 접근** — 인터랙티브 요소 키보드 도달 불가
- **빈 요소** — 텍스트 없는 button, a 태그
- **랜드마크** — 시맨틱 태그 부재
- **언어 속성** — html lang 누락/불일치
- **링크 목적** — 동일 텍스트 다른 대상

## 검증 절차

1. 사전 분석 결과를 기반으로 검사 스크립트 동적 작성
2. 모든 페이지/뷰/라우트에 대해 검사 실행 → 위반 목록 출력
3. **위반이 있으면:**
   - 각 위반의 impact(critical/serious/moderate/minor) 확인
   - 해당 요소의 소스 코드 수정
   - 재검사 실행
   - **위반 0건이 될 때까지 무한 반복**
4. 위반 0건 확인 후 추가 검증 진행

## 추가 검증 (자동화 도구가 못 잡는 항목)

자동 검사 통과 후, 프로젝트에 해당하는 항목만 Playwright/수동으로 추가 확인한다:

- 스크린 리더 탐색 순서가 논리적인지
- 포커스 트랩이 모달/다이얼로그에서 정상 작동하는지
- 동적 콘텐츠 변경 시 aria-live가 적절히 알림하는지
- 커스텀 위젯의 키보드 인터랙션 패턴이 WAI-ARIA Authoring Practices에 부합하는지
- 스킵 네비게이션이 작동하는지

해당 없는 항목은 건너뛴다.

합리적인 선에서 최대한 많은 서브에이전트를 병렬로 사용한다.

**검증:**
- `.claude/hooks/accessibility-validator.ps1`에서 자동 검증

**검증 실패 시:**
- 위반 사항 분석
- 접근성 문제 수정
- 검증 통과할 때까지 반복

서브에이전트는 항상 haiku를 사용한다.
